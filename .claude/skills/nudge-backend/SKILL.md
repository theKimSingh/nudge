# Nudge Backend Skill

Authoritative rules for all backend, data, auth, and security work in the Nudge React Native + Expo app. The app is destined for the App Store, so security and correctness are non-negotiable.

The user has chosen **Supabase JS only** as the data layer — no TanStack Query, no SWR, no Redux/Zustand. Don't suggest adding them.

---

## Folder layout (feature-based)

All backend code lives under `src/`. Create the directory the first time it's needed; never scatter backend code into `app/` or `components/`.

```
src/
├── lib/
│   ├── supabase.ts          # The single shared Supabase client
│   └── env.ts               # Typed env access via expo-constants
├── services/
│   └── <resource>.ts        # Pure async functions per table/domain
├── features/
│   └── <feature>/
│       ├── api/             # Feature-specific data hooks
│       ├── types.ts         # Feature TS types + Zod schemas
│       └── ...              # (screens, components — see frontend skill)
└── types/
    └── database.ts          # Generated by `supabase gen types typescript`
```

**Hard rule**: **never** call `supabase.from(...)`, `supabase.auth.*`, or `supabase.rpc(...)` directly inside an `app/` route file or a UI component under `components/` or `src/features/*/components/`. Always go through a service function or a feature hook.

## Supabase client setup (`src/lib/supabase.ts`)

Exactly **one** client instance for the whole app.

```typescript
import 'react-native-url-polyfill/auto';
import { createClient } from '@supabase/supabase-js';
import * as SecureStore from 'expo-secure-store';
import { AppState } from 'react-native';
import { env } from './env';

const ExpoSecureStoreAdapter = {
  getItem: (key: string) => SecureStore.getItemAsync(key),
  setItem: (key: string, value: string) => SecureStore.setItemAsync(key, value),
  removeItem: (key: string) => SecureStore.deleteItemAsync(key),
};

export const supabase = createClient(env.SUPABASE_URL, env.SUPABASE_ANON_KEY, {
  auth: {
    storage: ExpoSecureStoreAdapter,
    autoRefreshToken: true,
    persistSession: true,
    detectSessionInUrl: false,
  },
});

AppState.addEventListener('change', (state) => {
  if (state === 'active') supabase.auth.startAutoRefresh();
  else supabase.auth.stopAutoRefresh();
});
```

**Why these specifics:**

- `expo-secure-store` puts JWTs in iOS Keychain / Android Keystore. **Never** use `AsyncStorage` for tokens — it's unencrypted plaintext on disk and is an App Store / security review failure.
- `detectSessionInUrl: false` because mobile has no URL bar.
- The `AppState` listener is required by the Supabase docs. Without it, `autoRefreshToken` silently stops working when the app is in the foreground for long periods.

**SecureStore 2 KB limit fallback**: Expo SecureStore caps values at 2048 bytes. Supabase sessions can exceed this (especially with custom claims). If a user reports session loss or a write failure, switch to: encrypt the session with a key from `expo-crypto`, store the cipher in MMKV, and keep only the encryption key in SecureStore. Don't pre-implement this — wait until it's actually needed.

## Security (App Store hard requirements)

### Service-role key
**NEVER** import, embed, or reference the `service_role` key on the client. It bypasses RLS. Anything that needs service-role belongs in a **Supabase Edge Function** (`supabase/functions/<name>/index.ts`), called from the client via `supabase.functions.invoke('name', { body })`.

### Anon (publishable) key
The anon key is the only key that may live in the app bundle. Treat it as public — it'll be extracted from the IPA in seconds. All security comes from RLS, not from key secrecy.

### RLS — every table, every time
Every new table requires `alter table <name> enable row level security;` **and** at least one policy in the same migration. A table without policies returns zero rows to anon/authed users — which looks like a bug but is the database refusing the request.

Default policy template for user-owned rows:

```sql
create policy "owner can read"   on <t> for select using (auth.uid() = user_id);
create policy "owner can insert" on <t> for insert with check (auth.uid() = user_id);
create policy "owner can update" on <t> for update using (auth.uid() = user_id) with check (auth.uid() = user_id);
create policy "owner can delete" on <t> for delete using (auth.uid() = user_id);
```

Membership pattern (for shared resources):

```sql
create policy "members can read" on <t> for select using (
  exists (select 1 from <membership_table> m
          where m.<resource>_id = <t>.id and m.user_id = auth.uid())
);
```

Test policies before shipping: create two test users in the Supabase dashboard, run the same query as each, confirm isolation.

### Env vars
- Public config (Supabase URL, anon key) → `app.json` `expo.extra.*`, read via `expo-constants`.
- Local development overrides → `.env` (gitignored). A `.env.example` lives in the repo with the keys but no values.
- Production secrets → **EAS Secrets** (`eas secret:create`). Never commit a real `.env` for prod.

`src/lib/env.ts` validates at startup and throws if anything's missing — so the app fails fast at boot rather than mysteriously at the first network call:

```typescript
import Constants from 'expo-constants';
import { z } from 'zod';

const schema = z.object({
  SUPABASE_URL: z.string().url(),
  SUPABASE_ANON_KEY: z.string().min(1),
});

export const env = schema.parse(Constants.expoConfig?.extra ?? {});
```

### Input validation
Every user-supplied value crossing into Supabase passes through a Zod schema first. TypeScript types are compile-time only — they don't stop a malformed payload from a bad form, a stale build, or a tampered request. Zod is the runtime boundary.

Add `zod` to dependencies on first use, not preemptively.

## Data access patterns

### Service functions

Pure async functions. Take typed args. Return typed results. Always `.throwOnError()`.

```typescript
// src/services/profiles.ts
import { supabase } from '@/src/lib/supabase';
import type { Database } from '@/src/types/database';

type Profile = Database['public']['Tables']['profiles']['Row'];

export async function getProfile(userId: string): Promise<Profile> {
  const { data } = await supabase
    .from('profiles')
    .select('id, display_name, avatar_url, created_at')
    .eq('id', userId)
    .single()
    .throwOnError();
  return data!;
}

export async function updateProfile(
  userId: string,
  patch: Partial<Pick<Profile, 'display_name' | 'avatar_url'>>,
): Promise<Profile> {
  const { data } = await supabase
    .from('profiles')
    .update(patch)
    .eq('id', userId)
    .select()
    .single()
    .throwOnError();
  return data!;
}
```

**`.throwOnError()` is mandatory.** Without it, Supabase puts errors on `response.error` and the call silently "succeeds" — no caller will check. With it, errors throw and propagate to the UI's try/catch.

### Reads
- Always `.select('explicit, columns')`. Never `.select('*')` — leaks columns you didn't intend to expose, and inflates payload.
- Pagination on every list: `.range(from, to)` for offset, `.lt('created_at', cursor)` for keyset (preferred for infinite scroll).
- Indexes: any column used in `.eq()`, `.order()`, or `.lt()/.gt()` on a table expected to grow past ~1k rows needs an index. Add it in the same migration as the query that uses it.

### Mutations
- Return the row(s) modified: `.select().single()` or `.select()`.
- Multi-step / conditional logic that touches more than one table → write a Postgres function and call it via `supabase.rpc('fn_name', args)`. One round-trip, atomic, RLS still applies.
- Never auto-retry mutations.

### Realtime
Subscribe inside `useEffect` with cleanup. Scope the channel name to the feature + entity:

```typescript
useEffect(() => {
  const channel = supabase
    .channel(`profile:${userId}`)
    .on('postgres_changes',
        { event: 'UPDATE', schema: 'public', table: 'profiles', filter: `id=eq.${userId}` },
        (payload) => setProfile(payload.new as Profile))
    .subscribe();
  return () => { supabase.removeChannel(channel); };
}, [userId]);
```

## Auth flow

`src/features/auth/` owns the auth context. Mount one `<AuthProvider>` in `app/_layout.tsx`.

The context exposes:
- `session: Session | null`
- `user: User | null`
- `loading: boolean` (true until the initial `getSession()` resolves)
- `signIn(email, password)`, `signUp(email, password)`, `signOut()`

On mount: `supabase.auth.getSession()` to hydrate from SecureStore, then `supabase.auth.onAuthStateChange()` to keep state in sync.

**Route protection** uses Expo Router groups. Put authed screens under `app/(authed)/` with an `_layout.tsx` that reads `useAuth()` and `<Redirect href="/sign-in" />`s anonymous users. Don't sprinkle auth checks across individual screens.

**Sign-out** clears the cache (any in-memory state) and calls `supabase.auth.signOut()` (which clears SecureStore via the storage adapter).

## Error handling

- Service functions throw. UI catches with `try/catch`.
- Never surface a raw Postgres error to the user — translate to a user-friendly message at the UI boundary.
- Idempotent reads (GETs) may retry with exponential backoff, max 3 attempts, only at the service layer.
- Mutations never auto-retry — the user re-triggers.

## Types & migrations

After **every** schema change, regenerate types:

```bash
npx supabase gen types typescript --project-id <id> > src/types/database.ts
```

Schema changes go through migration files in `supabase/migrations/` (`supabase migration new <name>`). Never edit the prod schema in the dashboard for anything that needs to be reproduced. Local Supabase CLI for dev (`supabase start`), `supabase db push` to apply.

## What NOT to add

- **No** TanStack Query, React Query, SWR. The user explicitly chose Supabase JS only.
- **No** Redux, Zustand, Recoil, MobX. Local state via `useState`; cross-screen state via the auth context (or a feature-scoped React context if genuinely needed).
- **No** wrapper class around the Supabase client. A service function calling `supabase.from(...)` directly is correct — extra layers add maintenance with zero benefit.
- **No** new dependency without checking `package.json` first and asking the user before installing.

## Quick checklist before declaring backend work done

1. New table → RLS enabled + at least one policy in the same migration?
2. Service function → uses `.throwOnError()`?
3. Read → explicit columns, not `select('*')`?
4. List query → paginated?
5. Token storage → `expo-secure-store`, never `AsyncStorage`?
6. New env var → added to `app.json` extra **and** `.env.example` **and** validated in `src/lib/env.ts`?
7. User input → Zod-validated before hitting the DB?
8. Service-role key → not on the client?
9. Schema changed → `database.ts` regenerated?
